# Database Migration Instructions — Security Features

This guide walks through adding the security tables to your existing PostgreSQL database.

## Prerequisites

- PostgreSQL 12+ with existing `ia_vendedor` database
- Alembic already initialized in `backend/alembic/`
- Current user has superuser/migration permissions

## Steps

### 1. Update Alembic env.py to include security models

Edit `backend/alembic/env.py`:

```python
# In the imports section, add:
from app.core.database.security_models import Base as SecurityBase

# In the run_migrations_online() function, update target_metadata:
from sqlalchemy import pool

def run_migrations_online() -> None:
    # Get existing configuration
    configuration = config.get_section(config.config_ini_section)
    
    # Import all models
    from app.core.database.models import Base as ModelsBase
    from app.core.database.payment_models import Base as PaymentBase
    from app.core.database.security_models import Base as SecurityBase
    
    # Combine all metadata
    target_metadata = [ModelsBase.metadata, PaymentBase.metadata, SecurityBase.metadata]
    
    # ... rest of configuration ...
```

### 2. Generate migration

```bash
cd backend

# Generate autogenerated migration
alembic revision --autogenerate -m "Add security features: audit logs, 2FA, API keys"

# Review the generated file
cat alembic/versions/*_add_security_features.py
```

Expected migration file should create three tables:
- `audit_logs`
- `totp_2fa`
- `api_keys`

### 3. Review migration file

The generated `alembic/versions/XXX_add_security_features.py` should look like:

```python
"""Add security features: audit logs, 2FA, API keys

Revision ID: abc123def456
Revises: previous_revision
Create Date: 2024-01-XX XX:XX:XX.XXXXXX
"""

from alembic import op
import sqlalchemy as sa

revision = 'abc123def456'
down_revision = 'previous_revision'
branch_labels = None
depends_on = None

def upgrade() -> None:
    # Create audit_logs table
    op.create_table(
        'audit_logs',
        sa.Column('id', sa.String(50), nullable=False),
        sa.Column('seller_id', sa.String(255), nullable=False),
        # ... more columns ...
        sa.PrimaryKeyConstraint('id')
    )
    op.create_index('idx_seller_id', 'audit_logs', ['seller_id'])
    op.create_index('idx_event_type', 'audit_logs', ['event_type'])
    # ... more indexes ...

    # Create totp_2fa table
    op.create_table('totp_2fa', ...)
    
    # Create api_keys table
    op.create_table('api_keys', ...)

def downgrade() -> None:
    op.drop_table('api_keys')
    op.drop_table('totp_2fa')
    op.drop_table('audit_logs')
```

### 4. Apply migration

```bash
# Test migration in reverse
alembic downgrade -1
alembic upgrade +1

# If successful, apply to production
alembic upgrade head

# Verify tables created
psql -U ia_vendedor -d ia_vendedor -c "\dt"
```

Output should show:


## Troubleshooting

### Migration fails with "table already exists"

```bash
# Check if tables exist
psql -U ia_vendedor -d ia_vendedor -c "\dt audit_logs"

# If they exist and you want to drop them:
psql -U ia_vendedor -d ia_vendedor -c "DROP TABLE IF EXISTS audit_logs, totp_2fa, api_keys;"

# Then retry migration
alembic upgrade head
```

### Permission denied error

```bash
# Run as superuser
sudo -u postgres psql -d ia_vendedor -c "GRANT ALL PRIVILEGES ON SCHEMA public TO ia_vendedor;"
sudo -u postgres psql -d ia_vendedor -c "ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO ia_vendedor;"
```

### Connection refused

```bash
# Ensure PostgreSQL is running
sudo service postgresql status

# If not running:
sudo service postgresql start
```

### Memory issues with large dataset

```bash
# If migration is slow or times out, run in maintenance mode
psql -U ia_vendedor -d ia_vendedor -c "
  SET maintenance_work_mem = '4GB';
  ANALYZE;
"
```

## Performance Notes

### Index Strategy

The migrations create indexes on:
- `seller_id` - for tenant filtering
- `event_type` - for event type queries
- `created_at` - for date range queries
- `(seller_id, event_type, created_at)` - composite for complex queries

This balances read performance (quick queries) with write performance (indexes add overhead).

### Partitioning (Optional, for large deployments)

If audit logs grow very large (100M+ rows), consider partitioning by date:

```sql
-- Partition audit_logs by month
CREATE TABLE audit_logs_2024_01 PARTITION OF audit_logs
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE audit_logs_2024_02 PARTITION OF audit_logs
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Automatically remove old partitions:
SELECT partman.create_parent('public.audit_logs', 'created_at', 'native', 'monthly');
SELECT partman.run_maintenance();
```

## Next Steps

After migration succeeds:

1. **Update application**
   - Import security models in app startup
   - Add rate limiting middleware to endpoints
   - Add audit logging to request handlers
   - Implement 2FA endpoints
   - Implement API key endpoints

2. **Configure services**
   - Set environment variables (see `.env.example`)
   - Configure Redis for rate limiting
   - Configure email for 2FA/API key notifications

3. **Test security features**
   - Run `pytest tests/test_security.py`
   - Manual testing of 2FA flow
   - Manual testing of rate limiting
   - Manual testing of audit logs

4. **Deploy to production**
   - Follow production checklist in `SECURITY_FEATURES.md`
   - Set up monitoring/alerts
   - Document incident response procedures
